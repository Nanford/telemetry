# A-1-2 现场测试指南

> 目标:用一次现场巡检,把 A-1-2 每个垛位的**真实 SLAM 坐标**标定出来、填进系统,
> 让温湿度数据正确归属到垛位、在地图和巡检详情页正常显示。
> 适用设备:`go2_01` · 云端:`windoor.leenf.online`

---

## 0. 先懂一个原理(决定了流程为什么这么简单)

匹配是**后端按坐标现算的**(`backend/src/inspection-batches.js` 的 `matchInspectionPoint`):
对每条采样,算它到每个采集点的距离,`≤ radius` 且最近的就算命中该垛位。采集点坐标来自
`backend/src/config.js` 的 `slam.points`。

**关键推论:改后端采集点坐标 → 已经录进数据库的轨迹会被"追溯重新匹配",不用重走。**
所以今天的核心只有一件事:**拿到每垛真实坐标,填进 `config.js`,部署。**

> 今天(7-20)首测为什么 0 命中:采集点还是旧的 A1–A5(在正 y),而真机在负 y,
> 整条轨迹离最近采集点 2.41m > 半径 0.8m。坐标对不上 → 必然 0 命中。

## 1. 前置检查(5 分钟)

  ```bash
  python tools/calibrate_from_trail.py --fetch --device go2_01 --out test.png
  ```
  能拉到今天的数据、出图,就说明链路通(拉的是历史数据也没关系,验证命令能跑)。

## 2. 阶段一 —— 标定巡检走一趟(20–30 分钟)

1. **门口(右短边)开机 = 坐标原点。** 记一下开机的具体位置和朝向(拍张照),以后尽量每次一样。
2. 遥控进仓,走中央走道。
3. **按垛号顺序 `01 → 02 → … → 23`**,每到一个垛位:对准垛位、**原地停 ≥ 15 秒**,再走下一个。
   - 为什么按垛号顺序:工具按"停留先后"给垛位编号,顺序走 = 编号自动对上,省得事后对。
   - 为什么必须停够 15 秒:工具靠"停留时长"识别垛位、滤掉路过的一晃而过的点。停不够会被漏掉。
4. 走完 23 个垛位,回门口,停止遥控。

> 走位小贴士:上排(北)07→18、下排(南)06→01—缺口—23→19。按 01→23 走时,
> 会自然形成"下排一段 → 上排一段 → 下排一段"的蛇形,是合理路线。

## 3. 阶段二 —— 拉数据看识别(2 分钟,可反复)

```bash
cd Go2-SLAM-DEV/go2_env_agent
python tools/calibrate_from_trail.py --fetch --device go2_01 --out today.png
```

看终端输出:应识别出 **23 个停留簇**,每个打印质心坐标 `(x, y)` 和停留时长。
打开 `today.png`:23 个红圈应分成两排、位置合理、没有明显漏的。

---

## 4. 阶段三 —— 判断 & 调参 & 排错(不用重走!)

数据已经录在云上,**改参数就是本地重算,秒出**。只有"某垛真的没走到/停错"才需补走那一段。

| 现象 | 原因 | 处理(加参数重跑上面的命令) |
|---|---|---|
| 识别出 **少于 23 个** | 两个垛挨太近被并成一个 | `--cluster-r 0.35`(缩小簇半径) |
| 识别出 **少于 23 个** | 某垛停留太短没算上 | `--min-sec 8`(降低时长门槛) |
| 识别出 **多于 23 个** | 走道中途停顿被误判成垛位 | `--min-sec 20`(提高门槛) |
| 某个点坐标**明显离谱** | 该垛 SLAM 漂移 / 停错位置 | 只补走那一个垛位,重拉 |
| 拉不到数据 / 数量为 0 | device_id 错 / 后端 502 | 确认 `--device go2_01`、后端在跑 |

反复调 `--cluster-r` / `--min-sec`,直到稳定识别出 **23 个、位置合理**。

## 5. 阶段四 —— 生成真坐标

识别对了之后,生成坐标文件:

```bash
python tools/calibrate_from_trail.py --fetch --device go2_01 --emit-yaml app/config/points.A-1-2.yaml
```

然后把**终端打印的 23 个坐标**(或 `today.png` 图)发我,我把它同步进
`backend/src/config.js` 的 `slam.points`(前端地图 + 巡检详情页的匹配都读这里)。

> 需要人工核对一下:D1..D23 的编号是否和你实际走的垛号一致(按 01→23 走就应一致)。

## 8. 常见问题排查表

| 问题 | 排查 |
|---|---|
| 页面 502 打不开 | 云端 Node 后端没起,重启;nginx 反代是好的 |
| 部署后匹配还是 0 | ① `config.slam.points` 没更新或后端没重启;② 坐标符号/单位错(真机是负 y);③ radius 太小(垛间距 ÷2 要 > radius) |
| 读数串到别的设备 | 确认看的是 `device_id=go2_01`(库里还有 Go2 / Go2-SIM 的历史数据) |
| 详情页卡片还重叠 | 前端 `dist/` 没重新部署;清浏览器缓存 |
| 批次状态一直 `undetermined` | 缺"阈值规则":去规则页给 A-1-2 的点位/传感器配温湿度上下限,后端才能判正常/告警 |

## 9. 最终目标 —— 正常采集方式

标定一次后,日常巡检就是:**门口开机 → 遥控走垛位、每垛停一下 → 数据自动匹配上报 →
地图/详情页看温湿度归属到垛位**。坐标一次标定长期有效。

⚠ 前提:每次**开机位置和朝向尽量一致**。SLAM 原点=开机位姿,朝向偏差会让整套坐标旋转
(偏 3°、30m 处就偏 1.6m > 半径)。若发现某天匹配整体掉了,多半是开机朝向偏了——
届时再上"重定位 / 锚点校正"方案彻底解决(见项目记忆 A-1 floor layout)。
